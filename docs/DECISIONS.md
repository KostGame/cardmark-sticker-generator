# Decisions

## 2026-05-03: Начать с генератора стикеров

### Решение
Первая часть проекта — CardMark Sticker Generator. Приложение распознавания с камеры откладывается на будущий этап.

### Причина
Перед распознаванием нужен стабильный способ подготовить физические метки и JSON-манифест колоды.

### Последствия
Текущий репозиторий фокусируется на генерации, печати и экспорте. Распознавание не входит в текущий прототип.

## 2026-05-03: Статическое приложение для GitHub Pages

### Решение
Приложение должно быть статическим, работать из `index.html` и быть совместимым с GitHub Pages.

### Причина
Такой формат проще публиковать, проверять и использовать без установки серверной части.

### Последствия
Нельзя добавлять backend, серверные API или обязательный build pipeline без отдельного решения.

## 2026-05-03: Минимальная наклейка и подписи на подложке

### Решение
Сама наклейка содержит только машинно-читаемую метку и маленький ориентир верха. Название карты, ID, группа, стрелка и чекбокс остаются на подложке.

### Причина
После отделения наклейки на карте должна остаться минимальная метка без лишнего человеческого текста.

### Последствия
Печатный лист должен поддерживать подписи рядом с наклейкой, а не внутри основной области стикера.

## 2026-05-03: CardMark v0 является прототипным форматом

### Решение
CardMark v0 фиксируется как текущий прототипный формат, но не считается стабильным v1.

### Причина
Распознавание с камеры ещё не реализовано, поэтому формат может потребовать пересмотра после реальных тестов.

### Последствия
Изменения формата допустимы только через обновление `docs/CARDMARK_SPEC.md` и новую запись в `docs/DECISIONS.md`.

## 2026-05-03: Для MVP не использовать серверную часть

### Решение
MVP не должен использовать серверную часть.

### Причина
Генератор может работать полностью в браузере и не нуждается в хранении данных на сервере.

### Последствия
Все функции генератора должны работать локально или на GitHub Pages.

## 2026-05-03: Для MVP не использовать ИИ, OCR или внешние API

### Решение
MVP не должен использовать ИИ, нейросети, OCR или внешние API.

### Причина
Проект намеренно строится вокруг простых машинно-читаемых меток и локальной обработки.

### Последствия
Любая зависимость от AI/OCR/API требует отдельного решения и не входит в текущую цель.

## 2026-05-03: Для Таро использовать протекторы

### Решение
Для Таро рекомендуется клеить метки на протекторы, а не на сами карты.

### Причина
Карты Таро часто имеют коллекционную или личную ценность, и наклейки могут испортить поверхность.

### Последствия
Инструкция пользователя и документация должны сохранять эту рекомендацию.

## 2026-05-03: PDF через jsPDF является optional

### Решение
PDF-экспорт через `jsPDF` является optional, а браузерная печать остаётся обязательным fallback.

### Причина
CDN может быть недоступен, а браузерная печать обычно лучше сохраняет кириллицу и реальные размеры.

### Последствия
Любой PDF-экспорт должен иметь понятный fallback через «Печать / PDF».

## 2026-05-03: Изменения CardMark v0 требуют записи решения

### Решение
Любое изменение CardMark v0 требует обновления `docs/CARDMARK_SPEC.md` и записи в `docs/DECISIONS.md`.

### Причина
Формат метки станет контрактом между генератором, JSON-манифестом и будущим распознаванием.

### Последствия
Нельзя незаметно менять структуру 7×7, payload, checksum, диапазон ID или правила ориентации.

## 2026-05-03: Minimal local checks without a build step

### Решение
Добавить минимальную проверку проекта через Node script и команду `npm run check`, не превращая приложение в build-based проект.

### Причина
После PR-000 нужна воспроизводимая проверка пресетов, относительных путей, CardMark v0 и базового JSON/SVG экспорта.

### Последствия
Проект всё ещё работает как статическое приложение через `index.html`, но разработчики и агенты могут запускать `npm run check` перед PR.
